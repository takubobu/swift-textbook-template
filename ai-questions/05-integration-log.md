### Q1

**質問：**

なぜ struct ではなく classを使っているの？ 

**AIの回答の要点：**

なぜ struct ではなく class なのか？
結論から言うと、「SwiftData（データベース）のモデルとして機能させるため」です。
SwiftDataの @Model マクロは、クラスをデータベースのテーブルとマッピングします。これには以下の理由があります。

参照型（Reference Type）の必要性: データベースのレコードは「一意のID」を持ち、アプリ内のどこからアクセスしても「同じデータ」として扱われる必要があります。クラスは参照型であるため、メモリ上の同一インスタンスを複数のビューから参照し、同期を保つのに適しています。
変更の追跡: @Model は、クラスのプロパティが変更されたことを自動的に検知し、裏側のデータベース（SQLite）へ自動的に保存（永続化）します。この「変更通知」の仕組みは、Swiftのクラス構造と非常に相性が良いのです。
もし struct（値型）にしてしまうと、データを更新するたびに「コピー」が発生してしまい、データベースとの永続的な結びつきを維持するのが非常に困難になります。

**自分の理解：**

struct は計算など終わったら消えてもいいものの時に使用するのが適している。class はアプリを閉じても残しておきたいものの時に適している。
また、struct はコピーが渡されるのでアプリ内の「一覧画面」と「マップ画面」で同じレコードを見ているつもりでも、SwiftDataはそれが同じものだと判断できなくなる。

### Q2

**質問：**

PhotosPicker と UIImagePickerController の使い分け基準は？

**AIの回答の要点：**

結論から言えば、「特別な理由がない限り、すべて PhotosPicker を使う」のが正解です。

PhotosPicker は、「ユーザーが安全に写真を選んで渡す」ための、現代的でSwiftUIフレンドリーなツールです。
UIImagePickerController は、「カメラ撮影もライブラリ選択も全部まとめて面倒を見ていた、昔の大型ツール」です。

「写真をライブラリから選ばせたい」のであれば、PhotosPicker 以外を使う理由はほとんどありません。 逆に、UIImagePickerController を今からあえて使う理由は、「カメラ撮影という特定の機能と、レガシーなシステムへの対応がどうしても必要なとき」に限られます。

**自分の理解：**

写真を選ばせたい場合は PhotosPicker を使い、カメラを起動したい場合は UIImagePickerController を使う。
「PhotosPicker ＝ ギャラリーからの選択」、「カメラ ＝ 別の手段」という明確な役割分担で考えると迷うことがなくなる。
アルバムから写真を選ぶ」という要件であれば、迷わず PhotosPicker を選択する。

### Q3

**質問：**
保存処理のphotoData → PhotoRecord → modelContext.insertは、なぜ save() を呼ばずに保存される？ 失敗したらどうなる？

**AIの回答の要点：**

SwiftDataにおいて、save() というメソッドを明示的に呼ばなくてもデータが保存されるのは、SwiftDataが「オートセーブ（自動保存）」という仕組みを採用しているからです。

1.なぜ save() を呼ばなくていいのか？
SwiftDataは、「データの整合性」をシステム全体で一元管理しているからです。
メモリとディスクの自動同期: modelContext に対して insert() や delete() を行った際、それは即座にメモリ上の「変更の記録（変更トラッキング）」に反映されます。
適切なタイミングでの自動保存: SwiftUI のイベントループやアプリのライフサイクル（バックグラウンド移行時など）を監視しており、システムが「今ならディスクに書き込んでも安全だ」と判断したタイミングで、自動的にSQLiteデータベースへ書き込みを行います。
開発者の負担軽減: 開発者が「どこで保存ボタンを押すか」「保存の失敗をどう処理するか」を逐一制御する必要をなくし、ビジネスロジック（アプリの機能自体）に集中できるように設計されています。

2. もし保存に「失敗」したらどうなるのか？
ここが少し意外かもしれませんが、SwiftDataの自動保存は「失敗を隠蔽する」のではなく「失敗させない（または即時終了させる）設計」になっています。

**自分の理解：**

modelContext はメモリ上の変更を自動的にSQLiteへ書き込むため、手動の save() は不要で、書き込みに失敗した場合はシステム側でデータが安全な状態に保護（ロールバック）されるため、データが中途半端に壊れる心配はしなくていい。
