## 質問と回答の記録

### Q1

**質問：**

なぜstructではなくclassなのか？@Modelは具体的にどんなコードを自動生成している？

**AIの回答の要点：**

なぜ struct ではなく class なのか？
SwiftUIで頻繁に使われる struct ではなく、SwiftDataのモデルに class が使用される最大の理由は、「参照型（Reference Type）」である必要があるからです。

class を使う理由: データベース上の「唯一のレコード」と、アプリ内の「インスタンス」を1対1で結びつけ、変更を確実に同期するため。
@Model がしていること: クラスをデータベースで扱えるように拡張し、プロパティの監視、データベースへのマッピング、そして変更の自動同期を自動的にセットアップしています。

**自分の理解：**

データと画面を同期（シンクロ）させ続けるための仕組みとして、あえてclassとマクロという強力な武器を使っている

### Q2

**質問：**

@Query(sort: \.createdAt, order: .reverse)
普通の配列との違いは？ @State private var memos: [Memo] と書いたら何が起きる？

**AIの回答の要点：**

@Query は「データベース（SwiftData）と連動する生きた窓口」であり、@State は「メモリ上の変数」です。

**自分の理解：**

アプリを消しても残ってほしいかつ他の画面からでも同じデータを見たい場合はSwiftData (@Query)
今この画面で入力中の文字やボタンが押されているかのような、アプリ終了で消えても困るものの場合はUIの状態 (@State)

### Q3

**質問：**

save() を明示的に呼ばなくて良いのはなぜ？ いつディスクに書き込まれている？

**AIの回答の要点：**

SwiftData（およびCore Data）で save() を明示的に呼ぶ必要がないのは、「オートセーブ（自動保存）」の仕組みがフレームワークの深層に組み込まれているからです。

なぜ明示的な save() が不要なのか？
SwiftDataは、「モデルコンテキスト（modelContext）という作業場」と「ディスク上のデータベース（保存先）」を分けて管理しています。

1.変更の追跡（Autosave）:
    modelContext を介してデータを操作（挿入、更新、削除）すると、SwiftDataは「このデータに変更があった」ということを即座に検知します。
    
2.SwiftUIとの強力な連携:
    SwiftDataはSwiftUIのライフサイクルと密接に統合されています。アプリの実行ループの中で、SwiftDataは適切なタイミングを見計らって、「コンテキスト内の変更」を自動的にディスクへ書き出そうとします。

具体的に「いつ」書き込まれているのか？
実は、「この瞬間に書き込む！」という厳密な1つのタイミングがあるわけではなく、複数のトリガーによって保存が行われます。

**自分の理解：**

データの手動管理（手作業での保存・更新）から、宣言的なデータ管理（状態を定義するだけ）へ切り替える
プログラミング初心者ほど「自分で save() を呼ばないと消えるのではないか？」と不安になりがちですが、「フレームワークがよしなにやってくれる」ことを信じて、管理人に全投げするのが、SwiftDataの正しい付き合い方です。
